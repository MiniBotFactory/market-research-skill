# Jobs-to-be-Done (JTBD) 框架

JTBD 是一种用于理解用户真实购买动机的分析框架。核心主张：**用户购买产品，是为了完成某项"工作"（Job），而不是因为产品的功能或特性**。

---

## 核心概念

### 什么是"Job"

一个 Job 是用户在特定情境下试图达成的目标或解决的问题。

**Job 的三个维度**：

| 维度 | 描述 | 示例 |
|------|------|------|
| **功能性 Job** | 实际要完成的任务 | "整理和呈现销售数据" |
| **情感性 Job** | 用户希望感受到什么 | "感到自己掌控了局面" |
| **社交性 Job** | 用户希望在他人眼中如何呈现 | "被领导认为工作高效" |

**关键洞察**：用户通常为情感性和社交性 Job 付最多的钱，但描述需求时只说功能性 Job。

### 经典例子

> 哈佛教授 Clayton Christensen：人们购买奶昔，真正的 Job 是"打发开车通勤的无聊时间，让早上更好过"。竞品不是其他奶昔，而是香蕉和百吉饼。

## JTBD 访谈法：切换访谈（Switch Interview）

**最有效的 JTBD 数据来源**：访谈最近（3个月内）做出购买决定的用户，重建决策全过程。

### 切换访谈结构（45-60分钟）

**阶段 1：锁定时间（5分钟）**
```
"您是什么时候第一次开始使用[产品]的？"
"在那之前，您用的是什么？"
→ 目标：确定一个具体的切换时间点
```

**阶段 2：往前追溯——第一个想法（10分钟）**
```
"您还记得那次'哦，我要换个方式解决这个问题了'的瞬间是什么时候吗？"
"当时发生了什么让您有了这个想法？"
"在那之前您有多久没考虑过这件事了？"
→ 目标：找到触发决策的"第一个想法"时刻
```

**阶段 3：被动看待阶段（10分钟）**
```
"在您开始认真考虑之前，您有没有先观望了一段时间？"
"是什么让您一直没有行动？"
"什么事情让您觉得'现在必须解决这个了'？"
→ 目标：理解从"被动接受现状"到"主动寻找方案"的转变
```

**阶段 4：主动寻找阶段（15分钟）**
```
"您是怎么找到[产品]的？"
"您还考虑过哪些其他选择？"
"是什么让您最终选择了这个？放弃了哪些选择，为什么？"
→ 目标：理解竞争集（真实的竞品是什么）
```

**阶段 5：决策时刻（10分钟）**
```
"您最终做决定是在什么场合/情况下？"
"是您一个人决定的，还是需要和别人商量？"
"有没有什么让您犹豫的点？"
→ 目标：理解决策过程中的阻力和助力
```

**阶段 6：使用后回顾（5分钟）**
```
"现在使用这段时间后，您觉得最大的收获是什么？"
"有没有什么超出或低于您当初的预期？"
```

## Job Story 格式

JTBD 的输出使用 Job Story（而非传统 User Story）来描述需求：

**传统 User Story**（功能导向）：
```
作为 [用户角色]，我想要 [功能]，以便 [目的]。
示例：作为销售经理，我想要数据看板，以便查看团队业绩。
```

**Job Story**（情境导向）：
```
当 [情境/触发条件] 时，我想要 [动力/需求]，以便 [预期结果/消除焦虑]。
示例：当我周一早上要向总监汇报时，我想要立刻看到上周的完整数据，
       以便我不用临时整理 Excel 而显得准备不足。
```

**Job Story 的优势**：
- 包含情境（什么时候需要）
- 包含动力（为什么需要）
- 包含期望结果（用户真正在意的是什么）

## Job Map（工作流图谱）

将用户完成一个 Job 的全过程拆解为步骤，用于识别产品机会。

**Job Map 的 8 个通用阶段**：

| 阶段 | 用户在做什么 | 关键问题 |
|------|-------------|----------|
| **1. Define（定义）** | 确定要做什么 | 如何决定目标和要求？ |
| **2. Locate（定位）** | 找到需要的信息/资源 | 哪里可以找到所需的东西？ |
| **3. Prepare（准备）** | 整理和设置 | 如何为任务做好准备？ |
| **4. Confirm（确认）** | 验证准备就绪 | 如何确认可以开始执行？ |
| **5. Execute（执行）** | 实际执行任务 | 如何完成核心任务？ |
| **6. Monitor（监控）** | 检查进度 | 如何知道执行是否正确？ |
| **7. Modify（调整）** | 遇到问题时调整 | 需要改变什么？ |
| **8. Conclude（完成）** | 结束任务 | 如何知道任务完成了？ |

## 竞争集分析（从 JTBD 视角重新定义竞品）

JTBD 最大的价值之一：打破传统竞品边界。

**操作方法**：
从访谈中提取所有「切换来源」：

```
问题："在您开始用[我们产品]之前，您是怎么完成这件事的？"

可能的答案（真实竞品）：
- 用 Excel 手动整理 → Excel 是竞品
- 什么都没做，忍着 → "什么都不做"是竞品
- 雇了个实习生 → 人工劳动力是竞品
- 用其他 SaaS → 那个 SaaS 是竞品
```

**竞争集矩阵**：

```markdown
| 竞争对象 | 用户选择它的 Job | 我们需要做得更好的地方 |
|----------|-----------------|----------------------|
| Excel | 灵活可控，不需要学习成本 | 降低上手门槛，不输给 Excel 的灵活性 |
| 什么都不做 | 问题还不够痛 | 让"现状的代价"更显性 |
| [竞品SaaS] | [具体功能优势] | [需要弥补的差距] |
```

## JTBD vs. 传统用户研究对比

| 维度 | 传统用户研究 | JTBD |
|------|------------|------|
| 聚焦点 | 用户是谁（人口学特征） | 用户在什么情境下（动机） |
| 竞品定义 | 同类产品 | 任何能完成同一 Job 的方案 |
| 产品创新方向 | 用户要求的功能 | 帮用户更好地完成 Job |
| 最有价值的数据 | 用户调研/问卷 | 购买/切换/放弃时的访谈 |
| 适用阶段 | 优化现有产品 | 发现新产品机会、定义核心价值 |
